Personal study notes - how agents actually work behind the scenes.

0TL;DR

  • An agent = an LLM + tools + a loop + memory + a stop condition. It is the minimum scaffolding that turns a text-predictor into something that can pursue a goal.
  • Each scaffold patches one intrinsic weakness of the raw LLM (can't act, can't plan, can't remember, isn't reliably true).
  • The LLM is stateless. "Memory" is just a list of messages your loop keeps and re-sends in full on every iteration. Updating memory = appending to that list.
  • Context management = curating that list so it fits the window and stays relevant (append -> truncate -> summarize -> offload).
  • Tools are not in the OS. A tool is a framework function the model can only ask to call by emitting text; your framework is the bouncer that checks an allow-list before running it.
  • Agent-builder skills = decomposition, prompting, memory management, tool design, and evaluation/trace-debugging.

1Recap: the first principles of an LLM

Remember the distinction: first principles = irreducible facts (true even if LLMs never existed), not mechanisms (which are consequences). The six axioms:

  1. Language has statistical structure. Context constrains what is likely next (Shannon, 1948). The deepest one - if false, no LLM could exist.
  2. Meaning is revealed by context (distributional hypothesis; Firth/Harris, 1950s). Words in similar company have similar meaning.
  3. Anything can be turned into numbers. Symbols -> numbers makes text computable.
  4. A big enough adjustable function can approximate any pattern (universal approximation theorem, 1989).
  5. You can measure error and descend it with calculus. Score wrongness, follow the derivative downhill (gradient descent).
  6. Scale is available. Enough text (internet) + compute (GPUs). The only axiom that became true recently - why LLMs happened ~2020, not 1995.

The mechanisms fall out of the axioms (they are consequences, not principles):

AxiomMechanism it forces
1 - language is predictableNext-token prediction (the objective)
3 - turn into numbersTokens (a practical compromise)
2 - meaning is contextualEmbeddings + attention (Axiom 2 as arithmetic)
4 - approximate any patternThe Transformer (a huge parameterized function)
5 - measure & descend errorGradient descent / backprop (training)
6 - scale availableThe "Large" in LLM
One line

An LLM is what you inevitably get when you push "language is predictable + meaning is contextual" through "turn it into numbers, fit a huge function by descending error, at internet scale."

2What an agent really is

Strip away the hype: an agent is a loop around an LLM that lets it act, observe, and repeat. The canonical shape is the ReAct pattern (Reason + Act):

  1. A goal comes in.
  2. The LLM thinks: what should I do next?
  3. The LLM chooses an action: call a tool.
  4. The tool runs; its output (observation) is fed back into context.
  5. The LLM looks at the result and decides: am I done, or loop again?
  6. Repeat 2-5 until the goal is met, then answer.

The five building blocks: (1) the LLM (reasoning core), (2) tools, (3) the loop / control flow, (4) memory / context management, (5) a stop condition.

3Each agent part patches one LLM weakness

The most elegant way to remember the architecture: an agent is a set of scaffolds, each bolted onto a specific crack in the LLM.

LLM's intrinsic weaknessThe agent part that compensates
It only predicts text - can't act on the worldTools
It commits token-by-token; can't plan or self-correct mid-thoughtThe loop (act -> observe -> re-decide)
Its whole world is the context window - no memoryContext / memory management
It optimizes for plausible, not trueGuardrails + verification in the prompt/loop

4Is "agent = prompt + tools + loop" correct?

Mostly right - it is the correct skeleton. On a platform like Code Puppy the claim "designing a good agent is mostly writing the right prompt with the right guardrails" is ~80% true - which is exactly why building Caddie was just a JSON file: a prompt + a list of already-existing tools, dropped onto a loop the platform already runs. No loop or tool code written.

Where the word "just" undersells it - three load-bearing muscles the prompt can't fix:

  1. Tool design. Which tools you expose, and especially what their error messages say. "Error 500" vs "No rows for week 23 - try another week" is the difference between an agent that recovers and one that spirals.
  2. Context/memory management. On long tasks the window overflows; deciding what to keep/summarize/retrieve is a design axis the prompt alone doesn't solve.
  3. The loop's control logic. Stop conditions, retries, step limits, and verification live in the loop (code), not the prompt - because you can't fully trust the model to police itself.

Two quieter levers: model choice (a weak model won't follow even a perfect prompt) and evaluation (vibes lie; you must test).

Corrected one-liner

Agent = LLM + tools + loop + memory + stop condition. The prompt is the single biggest lever (especially where tools and loop already exist), but a good agent also depends on tool design, context management, and the loop's guardrails - the parts that compensate for what the prompt can't fix.

5The stateless LLM (the aha that unlocks everything)

The word "memory" tricks you into imagining the model remembering things. It doesn't. Every loop iteration is a brand-new, amnesiac API call. So where does memory live?

The key idea

Memory is a list of messages that YOUR code (or the framework) keeps, and re-sends in full on every loop iteration. "Updating memory" = appending to that list. There is no state inside the model - the state lives in your orchestrator loop.

The model gets the whole array, reads it fresh, emits one step, and forgets again. You append its step + the tool result, and hand the whole thing back next time. Once this clicks, "context management" is obvious: it is just curating that list so it (a) stays under the window size and (b) stays relevant.

6A full loop trace (worked example)

Task: "Compare my team's launches this week vs last week, find the biggest movers, and draft a leadership update."

Design first (the 4 decisions)

  1. Tools: list_files, read_file, run_python, write_file.
  2. Loop: ReAct - model either calls a tool or says DONE.
  3. Stop condition: model emits a final answer, OR 15 iterations max (safety).
  4. Memory policy: keep full transcript until big, then compact.

Now watch the messages list grow. Each iteration re-sends the ENTIRE list.

Iteration 0 - starting context sent to the model
// your code builds this list and sends it
[1] system: "You are an analyst agent. Tools: list_files, read_file,
             run_python, write_file. Think step by step. Say DONE when finished."
[2] user:   "Compare launches this week vs last, find movers, draft update."

// model responds with an ACTION, not an answer:
[3] assistant: think:"I need the report files first."
              action: list_files("~/reports")

// your code RUNS the tool, then APPENDS action + result. THIS is the memory update.
[4] tool:   "wk22_launches.csv, wk23_launches.csv"
Iteration 1 - resend the whole list [1..4]
[5] assistant: action: read_file("wk23_launches.csv")
[6] tool:   "<contents of THIS week's CSV ...>"
Iteration 2 - resend [1..6]
[7] assistant: action: read_file("wk22_launches.csv")
[8] tool:   "<contents of LAST week's CSV ...>"
// context now holds BOTH full CSVs - this is where it gets heavy
Iteration 3 - resend [1..8]
[9]  assistant: action: run_python("df_this - df_last, sort by delta")
[10] tool:    "Jordan +4, Priya +2, Dana -5, Chris -6 ..."
Iteration 4 - resend [1..10], model finishes
[11] assistant: action: write_file("update.md", "<draft>")
[12] tool:      "written"
[13] assistant: "DONE. Draft saved. Biggest movers: Dana -5, Chris -6 ..."
// stop condition (DONE) hit -> loop exits
What actually happened

The "memory" was just messages 1 -> 13, re-sent in full each pass, growing by append. The model never remembered anything; your loop did.

7Memory management: the two layers

Look at iteration 2 above: you stuffed two entire CSVs into the list. On a 40-step task that list explodes past the context window. So the orchestrator applies a memory policy. The moves, cheapest to fanciest:

  1. Append-only (do nothing). Fine for short tasks. Most agents start here.
  2. Truncation / sliding window. Drop oldest messages past a token budget. Risk: forgets the goal - so PIN the system prompt + task and only trim the middle.
  3. Summarization / compaction. When big, make a side LLM call to summarize the transcript, then REPLACE the old bulky messages with the summary. Two giant CSVs collapse to "this week vs last: Dana -5, Chris -6."
  4. Offload + retrieve (RAG). Don't paste raw data into the transcript; write it to disk / a vector DB, keep a pointer, and pull back only what you need when you need it.
  5. Structured scratchpad. Keep a separate tidy state object (e.g. findings.md) the agent updates and you inject fresh each turn - so important conclusions never get trimmed.
The two-layer model

Working memory = the message list inside the context window. Updated by APPEND each iteration; kept sane by TRUNCATE/SUMMARIZE.

Long-term memory = files / DB / vector store outside the window. Updated by WRITE; recalled by RETRIEVE. Survives across tasks and sessions.

An LLM only ever sees working memory. Long-term memory only helps if you RETRIEVE from it INTO working memory. (Caddie's long-term memory = the report file on disk; its working memory = the transcript; read_file is the bridge.)

8Compaction deep dive: summarize without losing the thread

The core question is: what is the "thread" that must survive? It is NOT the whole transcript - it is four specific things:

  • The goal (the original task)
  • Decisions already made (so it doesn't relitigate or contradict itself)
  • Facts learned that future steps depend on (numbers, file names, findings)
  • Open state (what's left to do)

Everything else - verbose tool dumps, dead-ends, chatter - is compressible or droppable. The rules:

  1. Pin the anchors - never summarize them. System prompt + original goal always injected verbatim. Drift happens when these get trimmed.
  2. Separate durable state from chatter. Keep a structured running scratchpad (learned / decided / open / next). Update it each loop, inject it fresh. Even if you drop raw history, conclusions never die. The single best trick.
  3. Summarize with a template, not free-form. Force structure: "Learned: ... Decided: ... Still open: ... Data lives in: wk23.csv". Loses far less than "summarize the above."
  4. Keep recent turns verbatim; compress only the old (rolling summarization). Recency carries the most signal.
  5. Preserve identifiers and numbers exactly; summarize the prose. Never let "Dana -5.3" become "Dana declined."
  6. Offload, don't delete - keep a pointer. Big outputs to disk; transcript keeps "full data in wk23.csv"; re-fetch if a later step needs it.
The one heuristic

For each piece of context ask: "Would a future step need this exact text?"
Yes -> keep verbatim, or offload + pointer.
Only needs the conclusion -> summarize.
Neither -> drop.

The trap

Compaction is itself an LLM call, so it can hallucinate or silently drop a key fact. That is why the durable scratchpad matters (it is deterministic, not summarized) and why you test your compaction on real runs.

9Tools, the OS, and permissions

A tool is not an OS feature. A tool is a function the FRAMEWORK exposes and makes callable by the model. Some tools wrap OS capabilities, but that wrapper is the point. Categories:

  • OS-backed: read_file, run_shell - leverage the OS, but through a framework function.
  • API-backed: query BigQuery, call Salesforce, hit a REST endpoint.
  • Compute: run_python.
  • Other agents: invoke_agent.
The whole safety story

The model never touches the OS directly. It only emits TEXT like read_file("x.csv"). The framework's code intercepts that text and decides whether and how to run it. That indirection is where all access control lives.

How agents get access - and how you control it

  1. The allow-list (main control). The agent config declares exactly which tools it may use - literally the "tools": [...] array in the agent JSON. This is least privilege: expose only what the job needs.
  2. The execution layer enforces limits. Sandboxing (restricted dirs, containers), read-only vs write scopes, argument validation, deny-lists (e.g. "never kill Teams on port 8080", "never touch ~/.code-puppy-venv").
  3. Human-in-the-loop approval. Dangerous tools (shell, delete, publish) can require confirmation BEFORE the framework executes them.
  4. Injection defense. Tool OUTPUTS are untrusted input (a file could say "ignore your instructions"). A good framework treats tool results as data, never as new permissions.
The warning that matters

An agent runs with YOUR OS permissions. The allow-list controls which doors exist, but a run_shell tool is a very wide door - it can do anything you can do on that machine. Real control = allow-list + sandbox + approval on the powerful tools. To truly lock down Caddie you would drop agent_run_shell_command (or gate it behind approval) so it can only read/write specific report files.

Mental model

The model can only ASK to use a tool by emitting text; your framework is the bouncer that checks the allow-list, validates the request, maybe asks you, then runs it. The tools array in the agent config is you handing the bouncer the guest list.

10Becoming an agent builder: the skill stack

Prompting and memory management are the two biggest levers, but they are two of five. In order of leverage:

  1. Decomposition / systems thinking. Turn a fuzzy goal into tool-sized steps. The meta-skill. Think "manager delegating to a brilliant but painfully literal intern."
  2. Prompting. Role, guardrails, output shape, examples. The single biggest dial.
  3. Memory / context management. What to keep, compress, offload, retrieve.
  4. Tool design. Which tools to expose, and what their outputs/errors say (a good error message is what lets an agent self-correct).
  5. Evaluation + trace-debugging. The underrated one. Test on real cases; READ the transcript to find where in the loop it went wrong.

On a platform like Code Puppy you need almost no heavy coding - the loop and tools are given, so prompt-craft feels like the whole game. The moment you build agents that run long, use custom tools, or chain other agents, decomposition + memory + eval start doing the heavy lifting.
