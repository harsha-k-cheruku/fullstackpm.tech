System Design from First Principles

How to make many machines behave like one reliable service. Not a catalog of architectures to memorize - the handful of forces (latency, failure, scale, distance) that make systems hard, and the small set of levers (caching, queues, replication, statelessness) that every design is assembled from.

Topic folder: first-principles-tutor/building · Captured 2026-07-12 · Cluster 3 of the Builder's Atlas; the applied face of Networks + Energy

0TL;DR

  • System design is complexity management at the scale of machines. One machine is simple; the moment you need more than one - for scale or survival - everything gets hard, because networks are slow and unreliable.
  • Everything is a tradeoff. There is no "best" design, only "best for these constraints." Name what you are spending (latency, consistency, cost, simplicity) to buy what you need.
  • The master tensions: latency vs throughput; consistency vs availability (CAP); cost vs performance; simplicity vs scalability.
  • Statelessness is the key to scaling. If a server holds no session state, you can run a thousand of them behind a load balancer. Push state to dedicated stores.
  • Caching is the universal speed (and cost) lever; queues are the universal decoupling and smoothing lever.
  • Failure is normal, not exceptional. At scale something is always broken; design for it with redundancy, retries, idempotency, and timeouts.

1Why systems get hard: one machine is easy, many is not

A single program on a single machine is comprehensible. You are forced past that by two pressures: scale (one machine cannot handle the load or store the data) and reliability (one machine is a single point of failure). Both force you to more than one machine - and that is where the pain begins, because the moment two machines must cooperate, they talk over a network.

The axiom: the network is slow, unreliable, and not free

Everything hard about system design flows from one fact: talking to another machine is orders of magnitude slower than talking to local memory, and the message can be delayed, duplicated, or lost. A memory access is ~nanoseconds; a network round-trip is ~milliseconds - a million times slower. And unlike a function call, a network call can just... not come back. Distributed systems are the discipline of getting reliable behavior out of unreliable, slow communication. (This is Networks where the edges have real, physical cost.)

2The master tradeoffs (there is no free lunch)

Every architecture is a set of choices among tensions. The senior skill is naming them explicitly instead of pretending one option is simply "better."

TensionWhat it meansYou spend one to buy the other
Latency vs throughputLatency = time for one request. Throughput = requests per second.Batching raises throughput but adds latency; optimizing one often costs the other.
Consistency vs availabilityEveryone sees the same data now vs the system always answers.The CAP tradeoff (§3) - can't fully have both when the network splits.
Cost vs performanceFaster/more-reliable almost always means more machines, memory, or money.Every "make it faster" has a bill; every "make it cheaper" has a risk.
Simplicity vs scalabilityA simple monolith vs a scalable distributed system.Distribution buys scale and pays in complexity - often the worst trade made too early.
Read vs write optimizationStructures that make reads fast often make writes slow (indexes, caches, denormalization).Know your read:write ratio before you optimize.
The one habit

For any design decision, say out loud: "This buys us X by spending Y." If you cannot name Y, you have not understood the decision - there is always a Y. A design with "no downsides" just has downsides you have not found yet.

3CAP: the unavoidable truth about distributed data

When data lives on multiple machines and they cannot all talk (a network partition - which will happen), you must choose between two options, and you cannot have both:

  • Consistency (C): every read sees the latest write - so during a partition you must refuse requests you cannot guarantee are current (sacrifice availability).
  • Availability (A): every request gets an answer - so during a partition you must serve possibly stale data (sacrifice consistency).
CAP in one sentence

When the network splits (P, which you don't get to opt out of), you must pick C or A. A bank balance chooses C (better to reject than show wrong money). A social feed chooses A (better to show a slightly stale feed than an error). Most real systems tune this per-operation. The practical descendant is eventual consistency: accept the write now, propagate it, and all copies agree "eventually" - the pragmatic default of the internet.

4Scaling: vertical vs horizontal

  • Vertical (scale up): a bigger machine. Simple, but there is a ceiling and it gets expensive fast - and it is still one point of failure.
  • Horizontal (scale out): more machines working together. Effectively unlimited and fault-tolerant, but forces you to solve distribution (load balancing, shared state, CAP).
Statelessness: the enabler of horizontal scale

Here is the linchpin idea. If a server keeps no per-user memory between requests (stateless), then every server is interchangeable - a load balancer can send any request to any server, and you can add or remove servers freely. The moment a server holds session state, requests must return to that server, and scaling breaks. So: push state out of your app servers into dedicated stores (databases, caches), and keep the servers stateless. This one principle is the foundation of nearly all scalable architecture. (It is state discipline at system scale.)

5Caching & queues: the two universal levers

Caching - store the answer so you don't recompute it

Caching is the space-time tradeoff at system scale: keep frequently-needed results in fast storage close to where they are used, so most requests skip the slow/expensive work. It is the single highest-leverage performance move in most systems.

The two hard caching problems

Invalidation ("one of the two hard things") - when the underlying data changes, stale cache entries lie. You must decide how/when to expire them. Staleness - a cache is a bet that slightly-old data is acceptable; sometimes it is not. Caching trades freshness for speed - name that trade every time.

Queues - decouple producers from consumers

A queue sits between "work arrives" and "work gets done." The producer drops a task and moves on; a consumer picks it up when ready. This buys three things: decoupling (the two sides can fail/scale independently), smoothing (a traffic spike fills the queue instead of crashing the worker - "backpressure"), and resilience (if a worker dies, the task waits). Queues turn synchronous, fragile chains into asynchronous, durable ones.

6State & storage: where the truth lives

Since servers should be stateless, state must live somewhere deliberate. Choosing the store is a first-principles match of data shape to access pattern:

StoreBest forTradeoff
Relational (SQL)Structured data with relationships; strong consistency; complex queriesHarder to scale horizontally; rigid schema
NoSQL (key-value, document)Massive scale, flexible schema, simple access patternsWeaker consistency; limited querying
Cache (in-memory, e.g. Redis)Ultra-fast reads of hot data; sessionsVolatile; must handle invalidation
Blob/object storageLarge files, images, model artifactsNot for querying; higher latency
Vector databaseSemantic search over embeddings (RAG)Approximate; specialized (see Designing AI Systems)
Normalize vs denormalize

A live version of the DRY tradeoff: normalized data stores each fact once (consistent, but reads need joins); denormalized data duplicates facts for fast reads (but risks drift, and writes must update copies). SQL leans normalized; large-scale NoSQL often denormalizes for read speed. It is the read-vs-write tradeoff made physical.

7Reliability: at scale, something is always broken

With one machine, failure is an event. With thousands, failure is a constant background condition - a disk, a network link, or a node is always down somewhere. So you design assuming failure, not hoping against it.

The reliability toolkit

Redundancy: no single point of failure - replicate every critical component so one death is survivable.

Timeouts: never wait forever on a network call; a hung dependency must not hang you.

Retries with backoff: transient failures are common; retry, but wait longer each time so you don't stampede a struggling service.

Idempotency: design operations so doing them twice is safe (retries + at-least-once delivery mean things will run twice). "Charge order #123" (idempotent) beats "charge $10" (dangerous if repeated).

Circuit breakers & graceful degradation: when a dependency is down, stop hammering it and serve a reduced experience instead of collapsing entirely.

Idempotency is the quiet hero

Because networks duplicate and retry, the safest distributed operations are ones that produce the same result whether run once or five times. Designing for idempotency removes a whole class of "it ran twice and double-charged" bugs. Remember this one - it is the most under-appreciated system-design idea, and it matters enormously for AI agents that retry tool calls.

8Edge & breaking cases

  • Premature distribution. The biggest system-design mistake: building microservices and distributed complexity before you have the scale to need them. A simple monolith is the right answer far more often than architecture blogs suggest. YAGNI applies to systems too.
  • The fallacies of distributed computing. Assuming the network is reliable, latency is zero, or bandwidth is infinite. It is not; every remote call is a slow, fallible thing.
  • Cache stampede. When a popular cache entry expires, thousands of requests hit the database at once. Real, and needs deliberate handling.
  • Distributed transactions are very hard. Keeping multiple stores perfectly in sync atomically is a deep problem; most systems avoid it via eventual consistency and idempotency rather than fighting for it.

9How to use it

The design-conversation checklist

1. Scale? How many users/requests/how much data - now and in a year? (Don't design for scale you'll never see.)

2. Read or write heavy? The ratio drives caching, indexing, and store choice.

3. Consistency needs? Does stale data hurt (money) or is it fine (feed)? Pick your CAP posture per operation.

4. What must not fail? Add redundancy, timeouts, retries, idempotency around the critical path.

5. What am I spending? Name the tradeoff for every choice. If you can't, dig until you can.
