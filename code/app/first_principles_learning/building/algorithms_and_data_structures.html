Algorithms & Data Structures That Matter

Not a competitive-programming grind. The small set of ideas a builder actually reasons with: how cost grows with size (Big-O), the space-time tradeoff, the four data structures you use daily, and vector similarity - the algorithm that powers modern AI retrieval. This is the Computation substrate made practical.

Topic folder: first-principles-tutor/building · Captured 2026-07-12 · Cluster 2 of the Builder's Atlas; the daily face of Computation as a Substrate

0TL;DR

  • An algorithm is a recipe; a data structure is how you arrange data so the recipe is cheap. They are two halves of one skill: making computation affordable.
  • Big-O measures how cost grows as input grows. It ignores constants and small inputs and asks the only question that matters at scale: does this blow up? O(1) and O(log n) are heaven; O(n^2) and worse are landmines.
  • The universal lever is the space-time tradeoff: you can almost always go faster by using more memory (caching, indexes, hash maps) - and vice versa.
  • Four structures cover most work: arrays/lists (order), hash maps (instant lookup), trees (sorted hierarchy), graphs (relationships).
  • Hashing is the workhorse behind instant lookup, deduplication, and caching expensive work (including model calls).
  • Vector similarity / nearest-neighbor is the algorithm under embeddings and RAG - how an AI system finds "the most relevant chunks" from millions.

1Big-O: the language of cost as it scales

The only performance question that survives changing hardware is: as the input grows, how does the work grow? Big-O answers exactly that, deliberately throwing away constant factors and small-input behavior to expose the growth shape. A laptop that is 10x faster does not save an algorithm whose cost quadruples every time the data doubles - scale always wins.

NotationNameDoubling the input...Everyday example
O(1)Constant...changes nothing. Heaven.Hash-map lookup; array index
O(log n)Logarithmic...adds one step. Nearly free.Binary search; balanced-tree lookup
O(n)Linear...doubles the work. Fine.Scanning a list once
O(n log n)Linearithmic...a bit more than doubles. The best sorting can do.Good sorts (merge/quick sort)
O(n^2)Quadratic...quadruples. Danger at scale.Nested loop over all pairs
O(2^n), O(n!)Exponential / factorial...explodes. Intractable.Brute-forcing all combinations
The direct link to the Computation substrate

This table is the tractable-vs-intractable wall in daily clothing. O(1) to O(n log n) are tractable - they scale. Exponential/factorial are the intractable problems (and the shadow of P vs NP): they are computable in theory but hopeless in practice past small inputs. When you hit exponential cost, stop hunting for a clever tweak - you have hit a wall, and the move is to approximate, cache, or change the problem.

The two honest caveats

Constants sometimes matter. Big-O hides the constant factor; for small, fixed inputs a "worse" O(n^2) with a tiny constant can beat an O(n log n) with overhead. Measure, do not assume. Average vs worst case differ. A hash map is O(1) average but O(n) worst case (many collisions). Know which case your data triggers.

2The space-time tradeoff: the universal lever

The single most useful practical idea in this note. You can almost always trade memory for speed. Precompute and store an answer (use space) so you do not have to recompute it (save time). Nearly every optimization is a version of this:

  • Caching - store the result of expensive work so repeat requests are instant (see System Design).
  • Indexes - a database index is a precomputed structure (usually a tree) that turns a full scan into a fast lookup.
  • Hash maps - spend memory on a table to buy O(1) lookup instead of an O(n) scan.
  • Memoization - remember results of a function so the same call is never computed twice.
Deep connection: this is Landauer + information theory

The space-time tradeoff is not just a trick - it echoes the information and energy substrates. Storing a precomputed answer is storing information to avoid re-doing the computation (which would cost energy/time). For an AI builder this is huge: caching a model's answer to a repeated prompt trades a little storage for skipping an expensive, slow, energy-hungry inference. The whole economics of AI serving is space-time tradeoffs.

3The data structures that actually matter

A data structure is a way of arranging data so the operations you care about are cheap. Choosing the right one is most of practical performance. Four cover the vast majority of real work:

StructureGreat at (cheap)Bad at (expensive)Reach for it when...
Array / listOrdered data; access by position O(1); iterate in orderSearching for a value O(n); inserting in the middleYou need order or a simple sequence
Hash map / dict / setLookup, insert, "have I seen this?" all O(1) averageAny notion of order; range queriesYou need to find things by key, fast
Tree (balanced / B-tree)Sorted order + fast O(log n) lookup and range queriesMore overhead than a hash map for pure lookupYou need sorted data or ranges (databases!)
GraphRepresenting arbitrary relationships/connectionsCan be expensive to traverse (O(V+E)+)Things relate to many other things (networks, deps)
The builder's move

Before writing a loop, ask: "what operation will I do most, and which structure makes that operation cheap?" Searching a list over and over is O(n) each time - put the data in a hash map and it is O(1). Ninety percent of "why is this slow?" is the wrong data structure, not a slow language.

4Hashing: the workhorse behind everything fast

A hash function turns any data into a fixed-size number (a "fingerprint"). The magic: the same input always yields the same fingerprint, and different inputs almost never collide. That one idea powers a startling amount of the stack:

  • Instant lookup (hash maps): use the fingerprint as an address, so you jump straight to the data instead of scanning - O(1).
  • Deduplication: same content → same hash, so you can spot duplicates instantly.
  • Caching: hash the request (or prompt) to make a cache key; if you have seen that key, return the stored answer. This is how you avoid repeating expensive AI calls.
  • Integrity & change detection: if the hash changed, the data changed (git, checksums, content-addressing).
For AI builders specifically

Hashing prompts is your cheapest optimization. Many requests to an AI system are identical or near-identical. Hash the normalized prompt; on a hit, serve the cached completion for free and instantly. This single technique can slash cost and latency (cross-ref Economics of AI). Note: hashing is exact-match; for semantic near-matches you need vector similarity (§7) - a "semantic cache".

5Trees & graphs: hierarchy and relationships

Trees model hierarchy (a file system, an org chart, the DOM) and, when balanced, give O(log n) search by keeping data sorted - which is exactly why database indexes are trees (B-trees). Divide the search space in half each step and a million items take ~20 comparisons.

Graphs model arbitrary relationships: nodes connected by edges. They are everywhere once you look - social networks, dependency graphs, maps, knowledge graphs, and the connections between your notes. The two core traversals are BFS (breadth-first: explore level by level, finds shortest paths) and DFS (depth-first: go deep then backtrack, good for exhausting possibilities).

The link to the Networks substrate

Graphs are the concrete, buildable form of Networks & Emergence. Network effects, hubs, and contagion from the Think wing are graph properties you can now actually compute over. Agent systems and RAG knowledge bases are graphs under the hood.

6Recursion & divide-and-conquer

Recursion is a function that solves a big problem by calling itself on smaller versions until a trivial "base case." It is decomposition made executable: "to solve this, solve two smaller copies and combine them." Divide-and-conquer (split in half, solve each, merge) is why the best sorts are O(n log n) - the "log n" is the number of times you can halve the problem.

The recursion gotcha

Recursion is elegant but each call uses memory (the "call stack"); too deep and it overflows. And naive recursion can re-solve the same subproblem exponentially (the classic slow Fibonacci) - the fix is memoization (the space-time tradeoff again) or converting to a loop. Elegant is not automatically efficient.

7Vector similarity: the algorithm that powers AI retrieval

This is the one algorithm every AI builder must understand, because it is the engine of embeddings, semantic search, and RAG. The chain of ideas:

  1. An embedding turns meaning into a vector - a list of numbers (a point in high-dimensional space). Text with similar meaning lands at nearby points. (This is the "compression of meaning" idea made into coordinates.)
  2. Similarity = closeness in that space. Measure it with cosine similarity (the angle between two vectors) or distance. Close vectors = similar meaning.
  3. Retrieval = nearest-neighbor search: given a query vector, find the closest stored vectors. Those are the most semantically relevant chunks.
  4. At scale you cannot compare against everything (that is O(n) per query over millions). So vector databases use Approximate Nearest Neighbor (ANN) indexes (e.g. HNSW - a navigable graph!) to get near-instant, approximately-best matches. Note the tradeoff: approximate buys enormous speed for a tiny accuracy cost - the space-time tradeoff yet again.
Why this unlocks RAG

Retrieval-Augmented Generation works because of this algorithm: embed your documents into vectors, store them in a vector DB, embed the user's question, find the nearest chunks, and feed those to the model as context. Vector similarity is literally how an AI system "looks things up." Understand this and RAG stops being magic - see Designing AI Systems.

8Edge & breaking cases

  • Premature optimization. "The root of all evil" (Knuth). Do not micro-optimize before measuring - most code is not the bottleneck. Get it correct and clear first, profile, then optimize the one part that matters. Big-O intuition tells you where the danger could be; a profiler tells you where it is.
  • Big-O lies about small n. For tiny inputs, the constant factors and simplicity dominate; a "worse" algorithm can win. Big-O is a scale question.
  • The curse of dimensionality. In very high-dimensional embedding spaces, distances get less meaningful and exact nearest-neighbor gets expensive - which is exactly why ANN (approximation) exists.
  • You rarely implement these from scratch. Languages and libraries give you hash maps, sorts, and vector DBs. The skill is choosing and reasoning about them, not reinventing them - which is why this note is about judgment, not code.

9How to use it

The four questions

1. What operation happens most? Pick the data structure that makes that cheap.

2. How does the cost grow with size? Get a rough Big-O. If it is quadratic or worse and the input can get big, redesign now.

3. Can I trade memory for speed? Cache, index, or memoize the expensive repeated work (especially AI calls).

4. Is this exact-match or similarity? Exact → hashing. Semantic → embeddings + vector similarity.
