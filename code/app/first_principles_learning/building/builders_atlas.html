The Builder's Atlas

The map of the second wing. The Think wing explains how the world works; the Build wing is the craft of making things that work in it. This note is the syllabus - what the craft contains, why each piece is load-bearing, and the order to learn it - all filtered through one question: what does a builder using AI actually need to reason with?

Topic folder: first-principles-tutor/building · Captured 2026-07-12 · The hub of the Build wing; the applied face of Computation

0TL;DR

  • Two wings. Think = understand the world from first principles. Build = turn that understanding into working systems. This is the entry to Build.
  • The whole craft has one enemy: complexity. Every fundamental here - abstraction, design principles, data structures, system architecture - is a tool for managing complexity so a human brain can still reason about the system.
  • Five deep-dives, in order: Coding Fundamentals → Design Principles → Algorithms & Data Structures → System Design → Designing AI Systems.
  • It is AI-centric on purpose. Every fundamental earns its place by mattering when you build with AI - the ruthless filter that keeps this from becoming a CS degree.
  • It is the Computation substrate, cashed out. Big-O is the tractability wall in daily life; system-design tradeoffs are thermodynamics + economics meeting real hardware.
  • The master skill is judgment under tradeoffs, not memorizing syntax. There is no "right" answer, only "right for these constraints" - the whole wing trains that instinct.

1Why a Build wing exists

The rest of this collection is the Think wing: substrates, technology, human behavior, economics, needs - all about understanding. But understanding without the ability to make is only half of being "a builder and a thinker." The Build wing is the other half: the craft of turning ideas into systems that actually run.

The ruthless filter

This wing is not a computer-science curriculum. A fundamental is included only if it passes two tests: (1) irreducible - it is a genuine first principle of building, not a passing tool or framework; and (2) load-bearing for an AI builder - you would actually reach for it while designing, building, or debugging a real AI-powered system. Everything that survives both tests is here; everything else (specific languages, frameworks-of-the-month, exam trivia) is deliberately out.

2The one big idea: it is all complexity management

Before the map, the single axiom the whole wing rests on. Software's hard limit is not the machine - machines are fast and patient. The hard limit is the human brain, which can hold only a handful of things at once. As a system grows, the number of interacting parts explodes, and understanding collapses.

The axiom under the entire craft

The primary job of every software fundamental is to manage complexity so that a limited human mind can still reason about the system. Abstraction hides complexity behind a simple surface. Decomposition splits it into pieces small enough to hold in your head. Design principles keep those pieces from tangling. Data structures and algorithms tame the complexity of cost. System design tames the complexity of scale, failure, and distance. Different tools, one enemy.

Keep this lens and every rule in the wing stops being arbitrary dogma and becomes an obvious move against complexity. That is the difference between a coder who memorizes rules and a builder who understands why they exist.

4The suggested track (the order matters)

The clusters build on each other - each is the next one's foundation, exactly like the Think-wing stack.

  1. Coding Fundamentals - abstraction, decomposition, state, interfaces. The atoms of thinking in code. Start here even if you already code; it names the instincts.
  2. Design Principles - DRY, YAGNI, SOLID, KISS, coupling/cohesion - derived from first principles, not memorized as commandments.
  3. Algorithms & Data Structures - Big-O, hashing, trees, graphs, recursion, and vector similarity (the algorithm under modern AI retrieval).
  4. System Design - the master tradeoffs (latency/throughput, CAP, cost), caching, queues, state, storage, reliability.
  5. Designing AI Systems - the capstone: the inference pipeline, RAG, context engineering, evals, and agents as distributed systems. Where Build meets the Technology track.
Two-week reading plan

Week 1 (code in the small): Coding Fundamentals → Design Principles → Algorithms & Data Structures. Week 2 (systems in the large): System Design → Designing AI Systems → then re-read the Technology track (AI, LLMs, Agents) with builder's eyes - it reads completely differently once you can see the systems underneath.

5Why exactly these serve AI building

Every note in this wing has a direct payoff the moment you build something with an LLM:

  • Coding Fundamentals - LLMs generate code; you must read and judge it. Abstraction and state are how you tell good generated code from a mess.
  • Design Principles - AI makes writing code cheap, which makes designing code the bottleneck. The principles are how you keep an AI-accelerated codebase from collapsing under its own speed.
  • Algorithms & Data Structures - vector similarity is literally how RAG finds relevant context; Big-O is how you reason about token cost and latency; hashing is how you cache expensive model calls.
  • System Design - an AI product is a distributed system with an expensive, slow, flaky component in the middle. Caching, queues, retries, and idempotency are what make it usable.
  • Designing AI Systems - ties it all together into the patterns (RAG, agents, evals) you will actually ship.

The unifying reframe

Thinking tells you WHAT to build and WHY; the craft tells you HOW to build it so it survives contact with reality. A great builder-thinker moves fluidly between the wings: uses the Think wing to choose the right problem, and the Build wing to solve it without drowning in complexity.

7How to use this wing

The three habits

1. Always ask "what complexity is this taming?" When you hit a rule or pattern, name the complexity it fights. If it fights none here, it is cargo-culting - drop it (that is YAGNI).

2. Always ask "what's the tradeoff?" There are no free wins in building - every choice buys one thing by spending another (speed for memory, simplicity for flexibility, consistency for availability). Name what you are spending.

3. Pair it with the Tutor prompt. Each note stands alone, but to truly own a topic, paste the Tutor super-prompt plus a note and have it quiz you to the edges.
