Coding Fundamentals from First Principles

Not a language tutorial. The handful of irreducible ideas that separate code a human can still understand from a tangle nobody dares touch: abstraction, decomposition, state, and interfaces - all in service of one fight, against complexity.

Topic folder: first-principles-tutor/building · Captured 2026-07-12 · Cluster 1 of the Builder's Atlas

0TL;DR

  • Code is written for humans, not machines. The machine will run anything; the constraint is the next person (often future-you) who has to understand it. Every fundamental serves that.
  • Abstraction = hide complexity behind a simple surface, so you can use a thing without holding all its details in your head.
  • Decomposition = break a big problem into pieces small enough to reason about one at a time.
  • State is the villain. Most bugs are the system being in a condition you did not expect. Minimize mutable state; prefer pure functions; make illegal states unrepresentable.
  • Interfaces/contracts = the promises between pieces. Depend on the promise, not the guts, and pieces can change independently.
  • The AI era raises the stakes: generating code is now cheap, so reading and judging it is the scarce skill. These fundamentals are your judgment.

1The axiom: the enemy is complexity, and the bottleneck is your brain

Start from the irreducible fact: a computer executes instructions flawlessly and tirelessly. It does not care if your code is ugly. So the true constraint on software is not the machine - it is the human working-memory limit. A person can hold only a few interacting things in mind at once. A system's difficulty is roughly "how many things must I understand simultaneously to safely change this?"

The one sentence

All coding fundamentals are techniques to keep the number of things you must think about at once small enough to fit in a human head. That is the axiom. Abstraction, decomposition, state discipline, and interfaces are four different moves toward the same goal. Judge any coding practice by asking: does it reduce what I must hold in mind, or increase it?

2Abstraction: hide detail behind a simple surface

Abstraction is giving something a simple name and interface so you can use it without knowing how it works inside. You drive a car with a wheel and pedals, not by thinking about combustion. A function named send_email() lets you send email without re-deriving SMTP each time.

Why it is the master tool

Abstraction lets you build in layers: once a detail is sealed behind a good name, your mind is free to think at the next level up. This is exactly the "mechanisms of one layer become the axioms of the next" idea from the Think wing - applied to code. Every powerful system is a tower of abstractions, each trusting the one below.

The leaky abstraction

Every abstraction leaks: the hidden details eventually surface (a network call abstracted as a simple function still fails when the network does). The skill is not pretending leaks do not exist - it is choosing abstractions whose leaks are rare and survivable. A bad abstraction hides the wrong things and forces you to fight it constantly. When you are working around an abstraction more than through it, it is the wrong one.

3Decomposition: divide until each piece fits in your head

If a problem is too big to reason about, split it into sub-problems, solve each, and compose them. A function should do one thing; a module should own one responsibility. The test: can you describe what this piece does in a single sentence without using "and"? If not, it is doing too much - split it.

Coupling and cohesion (the two dials)

Cohesion (want it HIGH): the degree to which the things inside one piece belong together and serve one purpose. High cohesion = a module that reads like it has a single job.

Coupling (want it LOW): how much one piece depends on the internals of another. Low coupling = you can change one piece without breaking others.

The whole art of structure is high cohesion, low coupling. Almost every design principle in the next note is a specific tactic for achieving these two.

4State: the real villain behind most bugs

Here is the deepest fundamental, and the least obvious. State is data that changes over time - variables, files, database rows, the contents of memory. The reason state is dangerous: the behavior of code that reads mutable state depends not just on its inputs but on the entire history of what happened before. That history is invisible, unbounded, and impossible to fully hold in your head.

Why most bugs are state bugs

A bug is usually "the system was in a state I did not anticipate." Shared mutable state is the worst: two parts of a program changing the same thing, in an order you did not predict (this is the heart of concurrency bugs - see System Design). The more mutable state, the more possible histories, the more the combinatorial explosion of "what could go wrong" outruns your brain.

The defenses (in order of power)

1. Pure functions. A function that only uses its inputs and only returns an output - no reading or changing outside state - is trivially understandable and testable: same input, same output, always. Push logic into pure functions; push state to the edges.

2. Immutability. Data that cannot change after creation cannot surprise you. Make new values instead of mutating old ones.

3. Make illegal states unrepresentable. Design your data so a wrong combination simply cannot be built (e.g. no "shipped order with no address"). The best bug is one the structure forbids.

4. Single source of truth. Never store the same fact in two places that can drift apart. Derive, do not duplicate.

5Interfaces & contracts: depend on promises, not guts

An interface is the promise a piece makes about what it does, separate from how it does it. If code depends only on the promise ("give me a user id, I'll give you their profile"), the implementation behind it can be rewritten, optimized, or replaced entirely, and nothing that used it needs to change.

This is the mechanism that lets systems scale

Stable interfaces are how large systems - and large teams - avoid collapsing into one tangled ball. Each piece is a black box behind a contract; you only need to understand the contracts, not every internal. It is abstraction and low coupling made concrete. (An API, from System Design, is just an interface between separate programs.)

6Naming & readability: the cheap superpower

"There are only two hard things in computer science: cache invalidation and naming things." A good name is an abstraction - it tells you what something does so you never open it. Readable code is a form of complexity management: it lets a reader reconstruct your intent without re-deriving it.

  • Name by intent, not implementation: active_users, not list2.
  • Code is read far more than written. Optimize for the reader; a few saved keystrokes are never worth a minute of a future reader's confusion.
  • The best comment explains WHY, not WHAT. The code already says what; comment the reason, the tradeoff, the gotcha.

7The AI era: writing got cheap, so reading got precious

The inversion that matters most now

For decades the scarce skill was producing code. LLMs collapsed that cost. So the bottleneck inverts (exactly the "verification < generation" story from Trust & Verification): the scarce skill is now reading generated code and judging whether it is correct, well-structured, and safe. These fundamentals are precisely that judgment - they are what let you look at an AI's output and see "this has hidden shared state" or "this abstraction leaks" or "this does three things."

  • AI writes plausible code, not necessarily correct code. It optimizes for looking right. Your fundamentals are the filter.
  • Give the AI good structure to work within. Clear interfaces and small, cohesive pieces make its output better and easier to check - you are decomposing the problem for it.
  • The human still owns the design. AI is superb at filling in a well-shaped hole and poor at deciding what the holes should be. Deciding the shape is this note.

8Edge & breaking cases

  • Over-abstraction. Abstraction has a cost: each layer is one more thing to learn. Abstracting too early, or for a case that never comes, adds complexity (see YAGNI in Design Principles). Abstract when the repetition or the need is real, not speculative.
  • Over-decomposition. Splitting into too many tiny pieces scatters the logic so you must jump around to follow anything. Cohesion can be destroyed by too much slicing, not just too little.
  • State is sometimes irreducible. The real world is stateful (a bank balance must change). The goal is not zero state - it is contained, explicit, single-sourced state, with pure logic around it.
  • Performance vs clarity. Occasionally the clear version is too slow and you must write uglier, faster code. Do it only where measured, and wall it off behind a clean interface so the ugliness does not spread.

9How to use it

The review checklist (run on any code - yours or an AI's)

1. Can I describe each function/module in one "and"-free sentence? (cohesion)

2. If I change this piece, what else breaks? (coupling - fewer is better)

3. What mutable/shared state exists, and could it be surprised? (state safety)

4. Am I depending on a stable promise, or on someone's internals? (interfaces)

5. Will the next reader understand my intent from the names alone? (readability)

The payoff

You cannot memorize your way to good code, but you can ask your way there. These five questions are complexity detectors. Code that answers them well stays soft and changeable for years; code that answers them badly calcifies into the thing nobody wants to touch.

10Glossary

TermMeaning
AbstractionHiding complexity behind a simple name/surface so it can be used without knowing its internals.
Leaky abstractionAn abstraction whose hidden details unavoidably surface (e.g. a "simple" network call that fails when the network does).
DecompositionSplitting a problem into pieces small enough to reason about individually.
CohesionHow well the contents of one module belong together (want high).
CouplingHow much one piece depends on another's internals (want low).
StateData that changes over time; the main source of unexpected behavior and bugs.
Pure functionA function whose output depends only on its inputs, with no side effects; trivially testable.
ImmutabilityData that cannot be changed after creation, so it cannot surprise you later.
Interface / contractThe promise a piece makes about what it does, separate from how; lets implementations change freely.
Single source of truthStoring each fact in exactly one place so copies cannot drift apart.
