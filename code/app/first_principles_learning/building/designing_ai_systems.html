Designing AI Systems

The capstone where the Build wing meets the Technology track. How to build a reliable product on top of a component that is non-deterministic, expensive, slow, and confidently wrong sometimes. RAG, context engineering, agents, and evals - all derived from that one strange constraint.

Topic folder: first-principles-tutor/building · Captured 2026-07-12 · Cluster 3 capstone of the Builder's Atlas; fuses with LLMs & Agents

0TL;DR

  • The one strange fact: your core component is a probabilistic function - given the same input it may answer differently, sometimes wrongly, at real cost and latency. Classic software assumes deterministic, cheap, instant, reliable functions. AI systems invert all four.
  • So the whole discipline is: build a reliable system around an unreliable core - by controlling its input (context), verifying its output (evals + guardrails), and managing its cost/latency.
  • Context is the real interface. You don't program an LLM with code; you program it with what you put in the context window. Context engineering is the new core skill.
  • RAG = give the model the right facts at answer-time via vector similarity retrieval, instead of hoping they're in its weights.
  • Agents = LLMs that loop, use tools, and hold state - which makes them distributed systems with a non-deterministic controller. Treat them accordingly (idempotency, retries, timeouts, observability).
  • Evals are non-negotiable. Without a way to measure quality, you cannot improve or safely ship a probabilistic system.

1The core problem: a strange new kind of component

Every technique in this note falls out of one shift. For 70 years, software was built from functions that are deterministic (same input → same output), cheap, fast, and reliable. An LLM call is the opposite on all four axes:

Classic functionLLM callConsequence for the builder
DeterministicProbabilistic - may vary; may be confidently wrongMust verify outputs; never fully trust; add guardrails
Cheap (~free)Expensive - real $ per call, scales with tokensCache, route to smaller models, minimize tokens
Fast (~instant)Slow - hundreds of ms to many secondsStream, parallelize, cache, set expectations
ReliableFlaky - rate limits, outages, timeoutsRetries, fallbacks, idempotency (system design)
The design mandate

Your job is to build a system that is more reliable, cheaper, and faster than the model in the middle of it. You do that by controlling what goes in (context), checking what comes out (evals/guardrails), and wrapping the flaky, costly call in the reliability toolkit from System Design. Every pattern below is an instance of this.

2The inference pipeline: what actually happens per request

To reason about cost and latency you must see the request path. A single AI feature is rarely one call - it is a pipeline:

user input
  -> preprocess / guardrail check (is this safe/valid?)
  -> retrieve context (RAG: embed query -> vector search -> fetch chunks)
  -> assemble prompt (instructions + context + query, within token budget)
  -> LLM call (the expensive, slow, probabilistic step)  <-- cache here!
  -> parse / validate output (is it well-formed? safe? grounded?)
  -> post-process / act (return, or call a tool, or loop)
Every stage is a design lever

Latency and cost are the sum across this pipeline, so each stage is a place to optimize: cache the retrieval, cache the LLM call (hash the assembled prompt - see hashing), trim the context to fewer tokens, run guardrails cheaply in parallel, and stream the output so perceived latency drops even if total time doesn't.

3Context is the real interface (context engineering)

You do not configure an LLM with code or parameters in the usual sense - you steer it almost entirely through what you place in its context window for that call. The context is the API. This reframes "prompting" from a phrase-crafting art into an information-assembly engineering problem:

  • The context window is a scarce, budgeted resource. It has a token limit and a cost per token, and models attend unevenly across it (the "lost in the middle" effect). So you must decide what earns a place in it - instructions, examples, retrieved facts, history - and in what order.
  • Garbage in, garbage out - literally. Most "the model is dumb" problems are actually "the context was wrong/missing/bloated." Fixing context beats swapping models most of the time.
  • Context engineering > prompt wording. Which facts you retrieve, how you compress history, what examples you include, and how you structure the whole payload matters far more than clever phrasing.
The reframe that makes you good at this

Treat the context window as a tiny, expensive working memory you are curating for the model on every call. The builder's question is not "what should I say?" but "what is the minimal, highest-signal set of information this model needs, arranged how, to get this right?" That is engineering, and it is the single highest-leverage AI-building skill. (This is the applied face of the Agents note's memory/compaction discussion.)

4RAG from first principles

Retrieval-Augmented Generation exists to solve a specific problem: a model only knows what was in its training data (frozen at a cutoff) and its weights are a lossy compression - it cannot reliably recall your private docs or fresh facts, and it will hallucinate to fill gaps. RAG's fix: don't rely on the model's memory; fetch the relevant facts at answer-time and put them in the context.

The RAG pipeline (built from earlier notes)

Index (once): split your documents into chunks → embed each chunk into a vector → store in a vector DB.

Query (per request): embed the user's question → nearest-neighbor search for the closest chunks → insert those chunks into the prompt as context → ask the model to answer using only that context.

The model now reasons over supplied facts rather than remembered ones - grounding the answer, enabling citations, and slashing hallucination.

Where RAG actually breaks

RAG is only as good as its retrieval. If the right chunk isn't retrieved, the model can't use it - and "retrieval quality" is a whole discipline (chunking strategy, embedding quality, re-ranking, hybrid keyword+vector search). Most failed RAG systems have a retrieval problem, not a model problem. The other trap: stuffing too many chunks bloats context, raises cost, and buries the signal. Retrieve precisely, not abundantly.

5Agents as distributed systems

An agent is an LLM placed in a loop: it can decide to call tools (search, code, APIs), observe the results, update its state, and decide the next step - repeating until done. This is powerful, and it quietly turns your AI feature into a full distributed system with a non-deterministic controller at the center. Everything from the System Design note now applies, with extra force:

  • Idempotency is critical. Agents retry tool calls; a non-idempotent tool ("send the email", "charge the card") can fire twice. Design tools to be safe to repeat.
  • Timeouts & loop limits. A probabilistic controller can loop forever or go off the rails. Cap steps, set budgets, add circuit breakers.
  • State & memory management. The agent's growing history overflows the context window; you must compact/summarize (space-time tradeoff on tokens).
  • Observability is mandatory. When a non-deterministic multi-step system fails, you cannot debug it without full traces of every step, prompt, and tool result.
  • Error compounding. In a chain of N steps each ~90% reliable, overall reliability is 0.9^N - it collapses fast. Keep chains short; verify between steps.
The sober take

Agents multiply both capability and failure surface. The reliability math (0.9^N) is why robust agent systems keep loops short, make every tool idempotent, verify aggressively between steps, and prefer constrained workflows over open-ended autonomy where correctness matters. Autonomy is a dial, not a switch - turn it up only as far as your verification can keep up.

6Evals: you cannot improve what you cannot measure

Because the core component is probabilistic, "does it work?" is not a yes/no compile check - it is a statistical question. Evals are the systematic measurement of output quality, and they are the difference between engineering and vibes.

The eval ladder

1. A golden dataset: a set of representative inputs with known-good outputs (or acceptance criteria). Your regression suite.

2. Scorers: exact-match/rules where possible; model-graded ("LLM-as-judge") for open-ended quality; human review for the hardest calls.

3. Run on every change: a new prompt, model, or retrieval tweak is scored against the golden set before shipping - so you catch regressions the probabilistic system would otherwise hide.

Why this is the deepest link to the Think wing

Evals are verification made operational, and building the golden set is applied epistemology ("how would I know this is good? what would falsify 'it works'?"). Since generation is cheap and verification is the bottleneck (the central AI-age asymmetry), your eval harness is arguably the most valuable thing you build - it is the moat, because anyone can call the same model, but few can reliably measure quality.

7Cost & latency as first-class constraints

Unlike classic software where compute is nearly free, here every call has a bill and a wait. Cost/latency are not afterthoughts - they shape the architecture:

  • Cache aggressively. Exact-match cache on hashed prompts; semantic cache (vector similarity) for near-duplicates. The cheapest call is the one you don't make.
  • Right-size the model. Route easy requests to small/cheap/fast models and only escalate hard ones to the frontier model (a "model cascade"). Most requests don't need the biggest brain.
  • Minimize tokens. Tokens are the unit of both cost and latency (and energy). Trim context, compress history, constrain output length.
  • Stream & parallelize. Streaming cuts perceived latency; independent calls should run concurrently, not in sequence.
The link to Economics of AI

These are the same forces from Economics of AI and the Energy substrate, at the level of a single system. A viable AI product is one whose per-request cost is below its per-request value - and caching, routing, and token discipline are how you get there.

8Edge & breaking cases

  • Over-engineering with AI. Not every problem needs an LLM, RAG, or an agent. Often a simple rule, a database query, or a classic algorithm is cheaper, faster, and more reliable. Reach for AI when the problem is genuinely fuzzy/linguistic - YAGNI applies hard here.
  • RAG when fine-tuning (or a bigger context) is better - and vice versa. RAG for changing facts and grounding; fine-tuning for changing behavior/format; long-context for one-off large inputs. Matching the tool to the need is the skill (cross-ref the LLMs note's ladder).
  • Prompt injection. Because context is the interface and it may contain untrusted text (retrieved docs, user input, tool output), attackers can smuggle instructions into it. Treat all context as potentially adversarial; never let retrieved text override system instructions or trigger unguarded actions.
  • Non-determinism in tests. You cannot assert exact output equality. Tests must check properties, ranges, or use eval scorers - a genuinely different testing mindset.
  • Model drift. Providers update models under you; a prompt tuned today can regress tomorrow. Your eval suite is what catches it - pin versions where you can.

9How to use it

The AI-system design checklist

1. Does this even need AI? If a deterministic method works, use it. Reserve the model for genuinely fuzzy problems.

2. What's the context strategy? What must be in the window, from where (RAG?), within what token/cost budget?

3. How will I verify the output? Guardrails + an eval harness with a golden set - build this early, not last.

4. Where are cost & latency spent? Cache, right-size the model, trim tokens, stream.

5. If it loops (agent), is it safe? Idempotent tools, step limits, timeouts, full traces, short verified chains.

The payoff

Anyone can call an API and get a demo. A builder ships something reliable, affordable, and improvable on top of an unreliable core - and that comes entirely from the disciplines here: control the input, verify the output, manage the cost, and measure everything. That is the whole craft, applied to AI.

TermMeaning
