LLMs from First Principles

Personal study notes - the irreducible facts an LLM is built on, and everything that falls out of them.

Topic folder: first-principles-tutor/llms · Captured 2026-07-11 · Companion: ../agents/agents_from_first_principles.html

0TL;DR

  • First principles = irreducible facts (true even if LLMs never existed). Mechanisms = consequences (tokens, attention, the Transformer). Keep the two piles separate.
  • There are 6 axioms. Five are old (1940s-1990s); only "scale is available" became true recently - which is why LLMs happened ~2020, not 1995.
  • An LLM is a next-token predictor: given text, it outputs a probability distribution over the next token, then samples one, repeatedly (autoregression).
  • It is built in stages: tokenize -> pretrain (predict next token on the internet) -> supervised fine-tune (follow instructions) -> preference-tune / RLHF (be helpful & safe) -> deploy with sampling.
  • It optimizes for likely, not true. The gap between those two words is where hallucinations live.
  • For your own business data, climb the ladder: prompting -> RAG -> fine-tune a small model -> (almost never) pretrain. RAG usually wins.

1Axioms vs mechanisms (the distinction that matters)

This is the correction that reframed everything. A "first principle" is an irreducible fact - something you cannot derive from anything simpler, and which would be true even if LLMs had never been invented. A "mechanism" (next-token prediction, tokens, attention) is a design consequence - a clever thing engineers do because the axioms are true.

Why it matters

If you only learn the mechanisms, you memorize how today's LLMs work. If you learn the axioms, you understand why any system like this must exist - and you can reason about what comes next. Mechanisms change; axioms don't.

2The 6 irreducible axioms

  1. Language has statistical structure. Words do not come randomly; context constrains what is likely next. (Claude Shannon, 1948.) The deepest one - if false, no LLM could exist.
  2. Meaning is revealed by context (the distributional hypothesis). Words used in similar company have similar meaning. (Firth / Harris, 1950s.) "You shall know a word by the company it keeps."
  3. Anything can be turned into numbers. Symbols -> numbers makes text a computable object. Without this, none of the math applies.
  4. A big enough adjustable function can approximate any pattern. (Universal approximation theorem, 1989.) A large parameterized function can fit essentially any relationship.
  5. You can measure error, and calculus tells you how to reduce it. Score the wrongness, take the derivative, step downhill. (Gradient descent = "roll downhill.")
  6. Scale is available. Enough text (the internet) + enough compute (GPUs) to do all of the above at enormous size. The only axiom that became true recently - the reason for the timing.

3The mechanisms that fall out

Every headline "how LLMs work" concept is just an axiom turned into engineering:

AxiomMechanism it forcesWhat it is
1 - language predictableNext-token predictionThe training objective: guess the next token.
3 - turn into numbersTokensText chopped into sub-word chunks, each an integer ID. A practical compromise between characters and whole words.
2 - meaning is contextualEmbeddings + attentionTokens become vectors; attention lets each token look at the others to build context-aware meaning. Axiom 2 as arithmetic.
4 - approximate any patternThe TransformerThe specific huge parameterized function (stacked attention + feed-forward layers).
5 - measure & descend errorGradient descent / backpropThe training loop that nudges billions of parameters to reduce prediction error.
6 - scale availableThe "Large"Billions of parameters, trillions of tokens, thousands of GPUs.
One line

An LLM is what you inevitably get when you push "language is predictable + meaning is contextual" through "turn it into numbers, fit a huge function by descending error, at internet scale."

4How an LLM is actually built (the real pipeline)

The mechanisms assemble into a concrete construction sequence. Each stage adds a different capability:

Stage 1 - Tokenizer

Learn a vocabulary of sub-word chunks (e.g. via byte-pair encoding) so any text can be mapped to a sequence of integer IDs and back. This is the numbers-from-text step (Axiom 3).

Stage 2 - Pretraining (the expensive one)

Show the model trillions of tokens of internet/text and train it on ONE task: predict the next token, over and over, via gradient descent. The output is a base model - it has absorbed grammar, facts, styles, reasoning patterns... but it is not helpful. Ask it a question and it may just continue with more questions, because it only learned "what text usually follows text."

Stage 3 - Supervised fine-tuning (SFT)

Continue training on a curated set of instruction -> good-response examples written by humans. Now the model learns the shape of being helpful: when asked a question, answer it. This turns a text-continuer into an instruction-follower.

Stage 4 - Preference tuning (RLHF / DPO)

Humans rank multiple candidate answers ("A is better than B"). The model is then tuned to prefer the kind of answers humans prefer - more helpful, honest, harmless. This is the "alignment" layer that makes it feel like an assistant rather than an autocomplete.

Stage 5 - Deployment (inference)

At use time the model runs autoregressively (Section 5), inside a fixed context window (Section 6), usually with a hidden system prompt setting its persona and rules. Nothing is learned at this stage - the weights are frozen; all "customization" now happens through the prompt, retrieved context, or tools.

Key takeaway

Pretraining gives knowledge and language; fine-tuning gives behavior. When you "prompt," you are steering a frozen model - you are not teaching it. Teaching only happens in stages 2-4, which you almost never do yourself.

5Autoregression & sampling

An LLM does not emit a whole answer at once. For each step it produces a probability distribution over the entire vocabulary (how likely each possible next token is), picks one, appends it, and feeds the longer text back in to pick the next. That is autoregression - it writes one token at a time, always conditioned on everything so far.

How it "picks one" is sampling, and you can control it:

  • Temperature - scales how sharp vs flat the distribution is. 0 = always take the single most likely token (deterministic, repetitive, good for facts/code). High (e.g. 0.9+) = flatter, more surprising choices (creative, but more errors).
  • Top-p (nucleus) - only sample from the smallest set of tokens whose probabilities sum to p (e.g. 0.9), cutting off the unlikely tail.
  • Top-k - only consider the k most likely tokens.
Why this explains a lot

Because generation is token-by-token sampling, the same prompt can give different answers, early tokens steer later ones (a bad first sentence drags the rest downhill), and "be more precise" often just means "lower the temperature."

6The context window

The model can only "see" a fixed maximum number of tokens at once - the context window. It is the model's entire world for that call. Everything shares it: the system prompt, the conversation history, tool outputs, retrieved documents, AND your new question.

  • It is finite. Overflow it and the oldest content must be dropped or summarized (this is exactly what agent memory management handles).
  • Attention is not uniform - "lost in the middle": models attend most strongly to the beginning and end of the window, and can gloss over material buried in the middle.
  • No persistence. Nothing outside the window exists to the model. Between calls it remembers nothing (see the agents notes - state lives in your code, not the model).
Practical rules

Put the most important instruction near the top or the very end. Don't bury the key fact in the middle of a giant paste. Long context is not free - relevance beats volume.

7Possible / true / likely (the triad)

The model is trained to produce the statistically likely continuation. "Likely" is used as a proxy for "true," bounded by what is "possible" to express in text - but the three are not the same:

WordMeaningThe model's relationship to it
PossibleAnything expressible in languageThe full space it can generate.
LikelyWhat commonly follows, per training dataWhat it actually optimizes for.
TrueWhat matches realityWhat we want - but it is not the objective.
The core caveat

The gap between "likely" and "true" is where hallucinations live. A confident, fluent, plausible answer is the model doing exactly its job (produce likely text) - which is why fluency is NOT evidence of correctness. Always verify anything that matters.

8Failure modes / breaking points

  • Hallucination. Plausible-but-false output. Root cause: optimizes for likely, not true (Section 7).
  • Lost in the middle. Ignores facts buried mid-context (Section 6).
  • Context overflow. Runs out of window; earliest content silently dropped.
  • Prompt sensitivity. Small wording changes swing outputs - because it is a statistical function of the exact tokens.
  • Shaky arithmetic / strict logic. It pattern-matches; it does not compute. Give it a tool for real math.
  • Knowledge cutoff. Frozen weights = no events after training; no live data without a tool/retrieval.
  • Sycophancy. Preference tuning can bias it toward agreeing with you rather than correcting you.
  • Sampling variance. Non-zero temperature = different answers to the same prompt.

9Prompting principles, derived from the mechanics

Good prompting is not folklore - each rule falls directly out of how the model works:

Because......you should
meaning is contextual (Axiom 2)Give rich, specific context; don't assume it knows your situation.
it optimizes for likely, not trueAsk it to show reasoning / cite / verify; treat fluency with suspicion.
early tokens steer later onesState the role and goal first; put constraints up front.
attention favors start and endPlace the key instruction at the top or bottom, not the middle.
it pattern-matches examplesUse few-shot examples to show the exact format you want.
temperature controls randomnessLow temp for facts/code, higher for brainstorming.
it can't do real math/live dataGive it tools (calculator, retrieval) instead of trusting it.

10The limits of first-principles thinking

First-principles reasoning is powerful, but it has real blind spots - know them so you blend it with other reasoning:

  • Emergence. Some capabilities appear only at scale and cannot be derived from the axioms (you had to build big models to discover them empirically).
  • Empirical facts you can't reason your way to. Which architecture wins, which data mix works - these are measured, not deduced.
  • Cost and time. Deriving everything from scratch is slow; analogy and precedent are often faster and good enough.
  • Wrong axioms. If a starting fact is subtly wrong, the whole tower is wrong - and confidently so.
  • Messy context. Pure first-principles can ignore real-world constraints (politics, legacy, budgets).
The blend

Use first principles to understand why, analogy to move fast, and empirical testing to find out what is actually true. First principles tells you what is possible and necessary; experiments tell you what is real.

11SLM vs general LLM: the decision ladder

"Should I build a small model on my business data?" Climb the ladder and stop at the first rung that solves your problem - each step up is far more expensive:

  1. Prompt engineering. Cheapest. Steer a frozen general model with a good prompt + system instructions. Try this first, always.
  2. RAG (retrieval-augmented generation). Keep the general model; at query time, fetch your relevant business data and paste it into the context. Best when the problem is "the model doesn't KNOW my data." Cheap, updatable, no training. Usually the right answer for business-knowledge use cases.
  3. Fine-tune a small model. Train a smaller open model on your examples when you need a consistent format/tone/domain behavior, lower latency, lower cost per call, or on-prem/privacy control. This changes behavior, not knowledge.
  4. Pretrain your own. Almost never. Only at massive scale with unique, huge, proprietary data and a serious budget. For nearly everyone this is the wrong rung.
Rule of thumb

Knowledge problem -> RAG. Behavior/format/cost/latency problem -> fine-tune a small model. Almost never pretrain. And always exhaust prompting before spending on either.

12Glossary / quick reference

TermMeaning
TokenA sub-word chunk of text with an integer ID; the unit an LLM reads and writes.
Next-token predictionThe core objective: given text, output a probability over the next token.
AutoregressionGenerating one token at a time, each conditioned on all previous ones.
EmbeddingA token/word represented as a vector of numbers capturing meaning.
AttentionMechanism letting each token weigh the others to build context-aware meaning.
TransformerThe neural architecture (stacked attention + feed-forward layers) behind LLMs.
PretrainingThe big next-token-prediction training that produces the base model (knowledge).
SFTSupervised fine-tuning on instruction/response pairs (instruction-following).
RLHF / DPOPreference tuning from human rankings (helpful, honest, harmless).
TemperatureSampling knob: low = deterministic/precise, high = random/creative.
Context windowMax tokens the model can see at once; its entire world for that call.
HallucinationPlausible but false output; the "likely vs true" gap.
RAGRetrieval-augmented generation: fetch your data at query time into context.
Fine-tuningFurther training that changes a model's behavior/format (not its knowledge).
